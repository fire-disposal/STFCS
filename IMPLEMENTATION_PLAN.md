## 技术栈选择

### 核心依赖
- **WebSocket**: 原生WebSocket API
- **状态管理**: Immer + 自定义订阅系统
- **事件系统**: RxJS（可选，简化版使用EventEmitter）
- **类型安全**: TypeScript（直接从后端导入类型）
- **构建工具**: Vite（现有）

### 可选依赖（根据复杂度决定）
- **RxJS**: 用于复杂的事件流处理（如果简化，使用EventEmitter）
- **Redux DevTools**: 用于状态调试
- **Pixi.js**: 现有渲染引擎（保持不变）
- **React Query**: 现有HTTP请求缓存（保持不变）

### 开发工具
- **测试**: Vitest + Testing Library
- **代码质量**: Biome（现有）
- **性能分析**: Chrome DevTools + 自定义监控
- **调试**: 自定义调试工具 + Redux DevTools

## 风险缓解策略

### 技术风险
1. **WebSocket兼容性问题**
   - 方案：使用WebSocket polyfill，测试不同浏览器
   - 回退：支持HTTP长轮询作为备选

2. **性能未达预期**
   - 方案：分阶段实施，每阶段进行性能测试
   - 回退：保留关键优化，移除次要优化

3. **类型同步问题**
   - 方案：直接从后端monorepo导入类型
   - 回退：手动同步类型定义，添加类型测试

### 项目风险
1. **开发时间超期**
   - 方案：优先实现核心功能，简化非关键特性
   - 调整：如果时间紧张，推迟阶段4的优化

2. **向后兼容问题**
   - 方案：使用适配器模式，支持并行运行
   - 测试：全面测试现有功能，确保无回归

3. **团队学习曲线**
   - 方案：提供详细文档和示例代码
   - 培训：组织技术分享和代码审查
