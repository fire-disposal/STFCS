# STFCS Colyseus 架构优化路线图

> 目标：把 Colyseus 作为唯一实时主线，逐步移除重复的网络、认证、状态与房间语义，实现“前端轻、后端薄、共享纯、规则集中”的可维护架构。

---

## 1. 这份路线图解决什么问题

当前项目已经具备比较完整的多人联机能力，但随着功能增长，出现了典型的“多套真相并存”问题：

- 客户端既有 `NetworkManager`，又有 `App` 的会话编排，还有 `GameView` 的房间生命周期逻辑。
- 前端同时存在 Redux slices、Zustand store、room.state 影子引用等多套状态来源。
- 服务端 `BattleRoom` 承担了认证、会话、房主、元数据、命令分发、心跳、对象创建等过多职责。
- 认证链路从“用户名”演化过来，但历史遗留的密码 UI、密码字段、密码参数还在部分页面中残留。

这些问题不一定会立刻引起故障，但会明显拉高后续开发成本：

- 新功能不知道应该放在客户端、房间类还是服务层。
- 调试多人状态时，很难确认“哪一层才是唯一真相”。
- 修改一处房间逻辑，容易波及前端 UI、认证流程、会话恢复和房间列表。

这份文档的核心目标是：**把 Colyseus 重新拉回架构中心，所有多人同步、房间生命周期、会话保活与状态更新都围绕它展开。**

---

## 2. 目标架构原则

### 2.1 一条主线

Colyseus 是唯一实时主线：

- 房间状态由 `Schema` 驱动。
- 多人同步依赖 `room.state` 与 `room.onStateChange()`。
- 房间发现依赖 Colyseus Listing/Matchmake 能力。
- 客户端进入、退出、重连依赖 Colyseus 房间生命周期。

### 2.2 一种真相

不同类型的数据只允许有一个权威来源：

- **共享游戏态**：只放在 Colyseus `Schema`。
- **本地 UI 态**：只放在客户端本地 store。
- **协议与枚举**：只放在 `contracts`。
- **规则与算法**：只放在 `rules`。

### 2.3 一层只做一件事

每个模块只承担一个明显职责：

- `BattleRoom`：编排房间生命周期与状态推进。
- `CommandDispatcher`：执行业务命令校验与状态修改。
- `AuthService`：管理用户名会话、保活、单会话约束。
- `NetworkManager`：管理连接、房间发现、房间创建/加入/离开。
- `GameView`：只做渲染与交互入口，不做房间生命周期。

### 2.4 以可开发性优先

优化的目标不是“更抽象”，而是“更好开发”：

- 新人打开代码时，能快速判断逻辑放哪里。
- 修改多人同步，不需要在多个状态系统之间来回跳。
- 新增一个房间机制，不必同时改前端多个模块和服务端多个入口。

---

## 3. 当前架构与理想形态的差异

### 3.1 当前状态

#### 客户端

- `App` 负责初始化、恢复用户、房间订阅、登录后切换视图。
- `NetworkManager` 负责连接、房间列表、房间创建/加入、短 ID、会话 token、心跳相关。
- `GameView` 仍承载大量交互与房间联动逻辑。
- `RoomSelector`、`RoomCard` 等大厅组件还存在历史密码语义残留。

#### 服务端

- `index.ts` 已经较瘦，但仍承担 HTTP 路由注册、Colyseus bootstrap 和测试用户创建。
- `BattleRoom` 仍比较厚：认证、会话绑定、房主转移、元数据刷新、消息接入、网络质量、阶段推进都在其中。
- `AuthService` 目前是用户名单会话模型，但还需要更明确的保活与过期清理边界。

#### 共享层

- `contracts` 与 `rules` 已经开始收敛，但仍需要严格保证“协议纯、规则纯”。

### 3.2 理想状态

#### 客户端

- `App` 只负责应用级状态切换和会话恢复。
- `NetworkManager` 只负责网络连接与房间操作。
- `GameView` 只负责把 room state 映射到 UI。
- 本地交互态与多人同步态完全分离。

#### 服务端

- `index.ts` 只做启动、注册、路由挂载。
- `BattleRoom` 只保留生命周期、消息接入、metadata 维护、状态推进。
- 认证、会话、房间占用、保活回收都由独立服务负责。

#### 共享层

- `contracts`：协议、DTO、版本、枚举。
- `rules`：纯算法、数值判定、静态数据。

## 4. 最终推荐分层

### 4.1 `packages/contracts`

只放：

- 游戏阶段、命令枚举、协议 payload、业务 DTO。
- 服务端与客户端共享的纯类型。

禁止：

- 任何运行时 Schema 类。
- 任何服务端副作用。
- 任何 UI 相关逻辑。

### 4.2 `packages/rules`

只放：

- 三阶段移动算法。
- 命中、伤害、护盾、象限判定。
- 静态规则表和武器/舰船规格读取。

禁止：

- `Room`、`Client`、`Schema`。
- React、Zustand、Redux。
- HTTP 访问、WebSocket 访问。

### 4.3 `packages/server`

只放：

- Colyseus 房间类。
- 认证与会话服务。
- 房间元数据与 HTTP 路由。
- 命令分发与服务端领域逻辑。

### 4.4 `packages/client`

只放：

- 渲染层。
- 本地 UI 状态。
- 房间连接、房间列表、登录会话恢复。
- 对 `room.state` 的消费与命令发送。

---

## 5. 理想的 `BattleRoom` 形态

`BattleRoom` 的理想定位是：**房间编排器，不是业务大脑。**

### 5.1 它应该负责什么

- `onCreate`：初始化状态、注册消息、设置 simulation interval、设置 metadata。
- `onAuth`：验证 token，确认用户名与会话是否有效，拒绝重复会话。
- `onJoin`：把已认证会话挂接到 room state。
- `onLeave`：标记离线、允许重连、释放占用。
- `onDispose`：清理房间绑定和定时资源。

### 5.2 它不应该继续负责什么

- 复杂的用户存储细节。
- token 生成与失效策略。
- 与房间创建无关的认证规则。
- 大量临时索引和业务副作用。
- 规则判定与战斗算法本身。

### 5.3 推荐拆出去的内容

- `AuthService`：用户名会话、心跳、单会话约束、房间占用。
- `RoomAccessPolicy`：允许进入/重连/踢出/占用规则。
- `RoomMetadataService`：name、phase、playerCount、ownerId 等展示字段。
- `RoomPresenceService`：记录当前 session 是否在线、是否心跳超时。
- `CommandDispatcher`：命令分发与业务更新。

## 6. 详细迁移路线

下面的迁移顺序，遵循一个原则：**先统一真相，再拆分职责，最后收口治理。**

---

### Phase 0：冻结边界，梳理现状

#### 目标

- 明确哪些数据属于 room.state，哪些属于本地 UI，哪些属于服务端会话。
- 清掉历史密码语义、旧登录语义、重复状态来源的残留。

#### 要做的事

- 梳理客户端所有状态来源：`room.state`、Redux、Zustand、localStorage、临时 ref。
- 梳理服务端所有状态来源：`GameRoomState`、`AuthService`、`matchMaker`、HTTP 路由。
- 标记所有密码相关字段、参数、UI、文案，统一改为用户名会话。
- 确认 Colyseus 房间列表、加入、离开、重连是唯一房间事实来源。

#### 完成标准

- 没有新的密码输入入口。
- 没有重复的房间真相。
- 没有不清楚归属的状态字段。

---

### Phase 1：统一会话与登录语义

#### 目标

- 用户名登录成为唯一认证入口。
- 同一用户名只能存在一个活跃会话。
- 会话有明确 TTL，超过 TTL 自动失效。

#### 要做的事

- 登录 API 只接收 `username`。
- 注册 API 如果保留，也只按用户名创建或补全用户，不再需要密码。
- 服务端建立用户名到 token 的单会话映射。
- `heartbeat` 接口周期刷新 `lastSeenAt`。
- `index.ts` 启动定时清扫任务，主动回收过期会话。
- 前端登录成功后保存 `auth_token`，恢复时先校验 token 再恢复会话。

#### 完成标准

- 用户 A 已登录后，另一个地方再次用 A 登录会被拒绝。
- token 过期或心跳中断后，自动清理登录状态。
- 前端刷新页面后能正确恢复或失效回到登录态。

---

### Phase 2：把 Colyseus 放回房间中心

#### 目标

- 所有多人游戏状态都以 room.state 为准。
- 房间生命周期由 Colyseus 管理，不由前端自行模拟。

#### 要做的事

- `NetworkManager` 只保留连接、创建、加入、离开、订阅列表。
- `App` 只做会话恢复、登录态切换、房间列表订阅切换。
- `GameView` 使用单独 hook 获取当前 room，不能自己持有房间生命周期。
- 房间列表采用 Colyseus realtime listing 或现有 matchmake 查询，不再让多个层同时持有房间缓存。

#### 完成标准

- 前端不存在第二套房间同步逻辑。
- `GameView` 不再关心 room 的生命周期细节。
- 房间列表显示与 Colyseus 房间状态一致。

---

### Phase 3：瘦身 `BattleRoom`

#### 目标

- `BattleRoom` 变成薄壳。
- 认证、会话、元数据、命令处理分离。

#### 要做的事

- 把 session 占用与释放逻辑交给 `AuthService`。
- 把 `roomOwnerId`、`playerCount`、`phase` 等元数据生成逻辑下沉到 `RoomMetadataService`。
- 把命令分发保持在 `CommandDispatcher`，不要再往 `BattleRoom` 里堆领域规则。
- 把 `registerMessageHandlers` 里重复的 try/catch 包装收敛成统一错误处理方法。
- 让 `BattleRoom` 仅保留少量房间级状态字段。

#### 完成标准

- `BattleRoom` 文件明显变短。
- 新增命令时，不需要同时改很多生命周期逻辑。
- 房间类不再承担“半个服务端”的职责。

---

### Phase 4：收敛客户端状态

#### 目标

- 本地 UI 态与多人共享态严格分离。
- 消灭影子状态和重复数据源。

#### 要做的事

- 只让 `room.state` 负责多人共享的玩家、舰船、阶段、房间元数据。
- 只让本地 store 负责选中对象、面板开关、视图缩放、临时交互模式。
- 删除或冻结所有“手动同步 room state 到本地 ref”的模式。
- 将 `GameView` 中可拆的小功能下沉为 hooks 或小组件。

#### 完成标准

- 不会出现 room.state 改了，但本地另一份状态没改的问题。
- 不会出现本地 store 和 room.state 同时表示同一个事实的情况。

---

### Phase 5：协议与规则收口

#### 目标

- `contracts`、`rules`、`server` 三者职责绝对清晰。

#### 要做的事

- `contracts` 只保留 DTO、枚举、Zod schema、协议常量。
- `rules` 只保留纯函数和规则数据。
- `server` 的业务逻辑只在服务端，不回流到客户端。
- 客户端不得直接依赖任何服务端运行时结构。

#### 完成标准

- 新增协议字段时，只需要改 `contracts`。
- 新增规则算法时，只需要改 `rules`。
- 新增房间行为时，只需要改 `server`。

---

### Phase 6：治理与防回归

#### 目标

- 防止以后又长出第二套状态管理或第二套认证语义。

#### 要做的事

- 增加依赖边界检查，禁止 client 依赖 server runtime。
- 增加循环依赖检查。
- 为 `BattleRoom`、`NetworkManager`、`GameView` 设定复杂度上限。
- 对房间加入、离开、心跳失效、重连恢复、房主删除房间等关键路径加测试。
- 对大厅 UI 中的“私密房间密码”等旧语义彻底清零。

#### 完成标准

- 新人能很快判断文件职责。
- 没有“临时先这么写，后面再改”的长期遗留。
- 任何新增逻辑都能自然落到明确层级。

---

## 7. 推荐的执行顺序

下面是最稳妥的实际推进顺序：

### 第 1 步：统一会话模型

- 用户名登录。
- 单会话约束。
- 心跳 TTL 回收。
- Colyseus `onAuth` 校验 token。

### 第 2 步：删掉所有密码语义

- 清理前端密码输入、密码弹窗、密码参数、相关文案。
- 清理后端密码字段与密码比较逻辑。

### 第 3 步：收敛房间生命周期

- `GameView` 通过 hook 读取当前 room。
- `NetworkManager` 只做网络与房间列表。
- `App` 只做会话切换。

### 第 4 步：瘦身 `BattleRoom`

- 把房主、会话、元数据、错误处理各自下沉。
- 把 room 类维持在“薄壳”。

### 第 5 步：进一步切分领域服务

- `AuthService`
- `RoomMetadataService`
- `RoomAccessPolicy`
- `PresenceService`
- `CommandDispatcher`

### 第 6 步：补治理与测试

- 边界 lint。
- 回归测试。
- 文档更新。

---

## 8. 哪些反模式要坚决避免

- 在客户端再写一套“房间缓存真相”。
- 在 `GameView` 里继续堆 room 生命周期、副作用、按钮逻辑、网络调用。
- 在 `BattleRoom` 里继续堆认证、会话、规则、统计、对象工厂。
- 让 `contracts` 继续增长成一个混合包。
- 让 `rules` 变成一个依赖 server 的“伪共享层”。
- 让密码、用户名、token、sessionId、shortId 混成不清楚的身份模型。

---

## 9. 验收标准

当架构真正接近理想形态时，应该能满足下面这些条件：

- `BattleRoom` 文件短而清晰，职责一眼能看懂。
- `GameView` 更像渲染容器，而不是应用层。
- `NetworkManager` 不再像一个小型后端。
- `AuthService` 能独立说明“谁可以登录、谁还活着、谁在房间里”。
- 所有多人共享状态都能在 `room.state` 找到。
- 不再有密码 UI、密码参数、密码登录这类旧语义。
- 一个用户名在同一时刻只能对应一个活跃会话。
- 心跳中断后，登录状态会自动回收。

---

## 10. 最后结论

如果要用一句话概括这次重构方向：

> **把 Colyseus 从“项目里的一个库”升级为“整个多人系统的唯一事实源”。**

这样做之后，客户端会更轻、服务端会更薄、共享层会更纯，整个项目的可维护性和可开发性都会明显提升。
