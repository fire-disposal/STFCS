# STFCS 游戏策划分析

> 从桌游设计角度审视 STFCS 的机制、流程和系统边界。
> 不讨论代码，只讨论"玩游戏时发生了什么"。

---

## 1. 核心问题

当前 STFCS 有**三个系统纠缠在一起**：

| 系统         | 定义                   | 当前状态                             |
| ------------ | ---------------------- | ------------------------------------ |
| **回合系统** | 谁能在什么时候做什么事 | 与阶段绑定，与 faction 联动          |
| **派系系统** | 谁和谁一伙，行动顺序   | 硬编码 2 个 faction，GM 不算 faction |
| **GM 系统**  | 主持人如何控制游戏     | `force_end_turn` 身兼数职            |

纠缠的后果：无法干净地加入新功能（世界观地图），也无法简化现有流程。

---

## 2. 重新定义：谁在玩游戏？

一场 STFCS 游戏涉及两种角色：

### GM（主持人）

- **目标**：讲述故事，设置挑战，裁决规则
- **操作**：部署 NPC 舰船，控制 NPC 行动（替代 faction 轮流），判定特殊情况
- **工具需求**：
  - 自由编辑任意舰船
  - 随时推进/暂停/回溯战斗
  - 控制多个 NPC 舰船（作为独立 faction 或单舰行动）
  - 设置胜利条件和突发事件

### 玩家

- **目标**：驾驶自己的舰船赢得战斗
- **操作**：移动、射击、管理辐能、使用特殊能力
- **工具需求**：
  - 清晰知道轮到自己时能做什么
  - 看到自己和敌人的状态

## 3. 核心游戏循环

### 单场战斗的生命周期

```
┌─────────────────────────────────────────────────┐
│                  GM 准备阶段                       │
│  • 选择/创建场景                                 │
│  • 放置双方舰船                                  │
│  • 设置地图地形（可选）                           │
│  • 说明胜利条件                                  │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│                  部署阶段                         │
│  • 玩家逐个放置舰船（或按预设位置）               │
│  • GM 放置 NPC 舰船                              │
│  • 所有玩家就绪后开始                             │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
             ┌───────────────┐
             │               │
             ▼               ▼
     ┌──────────────┐  ┌──────────────┐
     │ 派系 A 回合   │  │ 派系 B 回合   │
     │ (通常是玩家)  │  │ (通常是 NPC) │
     │              │  │              │
     │ 一艘一艘操作  │  │ GM 操作 NPC  │
     │ 每艘: 移动→   │  │ 舰船         │
     │ 攻击→排散     │  │              │
     └──────┬───────┘  └──────┬───────┘
            │                 │
            └──────┬──────────┘
                   ▼
            ┌──────────────┐
            │  回合结算     │
            │ • 辐能消散    │
            │ • 武器冷却    │
            │ • 护盾维持费  │
            │ • 过载恢复    │
            │ • 地形效果    │
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │  继续/结束?   │
            │              │
            │ GM 判断：     │
            │ • 一方全灭 →  │
            │   战斗结束    │
            │ • 达成目标 → │
            │   战斗结束    │
            │ • 继续 →     │
            │   下一回合    │
            └──────────────┘
```

### 关键决策点

1. **谁决定什么时候结束回合？** → GM。不是系统自动。
2. **谁控制 NPC 舰船？** → GM。不是"派系 B 的玩家"。
3. **多个玩家在同一个派系时怎么操作？** → 每人控制自己的舰船，派系内无顺序。
4. **为什么需要 faction？** → 仅用于定义行动顺序。谁先动，谁后动。

---

## 4. 派系系统的真实用途

派系的唯一用途：**定义行动顺序**。

```
回合 1: 派系 A（玩家联盟）→ 派系 B（命运之握）
回合 2: 派系 A（玩家联盟）→ 派系 B（命运之握）
...
```

但实际跑团中，情况更多样：

| 场景                | 派系设置                                                  |
| ------------------- | --------------------------------------------------------- |
| 4 玩家 vs 海盗      | 派系 A = 4 玩家（同一派系），派系 B = 海盗（GM 控制）     |
| 2v2 PvP             | 派系 A = 2 玩家，派系 B = 2 玩家                          |
| 3 玩家 + 中立空间站 | 派系 A = 3 玩家，派系 B = 中立站（可互动），派系 C = 敌舰 |
| 乱斗                | 每个玩家独立派系，各自为战                                |

**结论**：派系应该是**动态配置的数组**，最多支持 4-6 个，每个派系有：

- 名称和颜色
- 所属玩家（可能有多个，也可能无——GM 控制）
- 行动顺序索引

---

## 5. GM 的双重角色

GM 在游戏中扮演两个角色：

### 角色 1：规则仲裁者

- 推进战斗流程
- 裁决争议
- 设置场景

### 角色 2：NPC 指挥官

- 控制所有非玩家舰船
- 代表 NPC 派系做决策
- 有时代表中立势力

**这两个角色应该分开对待**：

```
GM-as-Arbiter:  永远存在，永远拥有最高权限
GM-as-Commander: 只有在控制 NPC 派系时才需要
```

当前的 HOST 角色同时覆盖了这两个角色。这本身没有错，但 `force_end_turn` 让 GM-as-Arbiter 和 GM-as-Commander 的职责混在一起。

## 6. 简化方案

### 方案 A：保持现状 + 接口清理

不改派系系统，只改 GM 接口：

```
GM 操作面板:
├── [流程控制]
│   ├── 推进到下一派系     ← 替代 force_end_turn
│   ├── 立即结束当前派系   ← 跳过剩余舰船
│   └── 回到部署阶段
├── [NPC 操作]
│   ├── 当前派系 = NPC → GM 逐艘操作
│   └── 当前派系 = 玩家 → 等待玩家操作
└── [编辑]
    ├── 编辑任意舰船
    ├── 添加/删除舰船
    └── 修改地图
```

**优点**：改动最小，现有代码几乎不动。
**缺点**：派系仍硬编码，世界观地图仍无法干净集成。

### 方案 B：双层状态机（见 phase-system-rethink.md）

```
mode = TACTICAL_COMBAT
turn = { turnCount: 1, activeFactionIndex: 0 }
factions = TURN_ORDER  // 可动态配置
```

**优点**：干净，可扩展，世界观地图可自然成为第二种 mode。
**缺点**：需要重构 phase/activeFaction 的使用代码。

### 推荐：方案 B 的温和版

仅做三点改动：

1. **`GamePhase` 改为 `GameMode`**——只表达"当前在干什么"
   - `DEPLOYMENT` / `COMBAT` / `WORLD`（将来）
2. **`TurnState` 结构**——表达"当前轮到谁"

   ```typescript
   turn: {
     number: number,       // 回合数
     factionIndex: number, // 当前派系在 TURN_ORDER 中的索引
   }
   ```

   `activeFaction` 改为纯函数推导，不再是状态字段。

3. **`TURN_ORDER` 动态化**——不再是常量数组
   ```typescript
   // 房间创建时由 GM 指定
   room.turnOrder = ["players", "pirates", "neutrals"];
   ```
   每个派系有 `playerIds` 数组（谁控制此派系的舰船）。

## 7. 世界观地图的位置

世界观地图**不是**对现有系统的修改，而是**一个全新的 mode**。

```
mode = WORLD_EXPLORATION
  │
  ├── 无 turn（战术回合在此模式下无意义）
  ├── 无 faction（玩家作为一个团队行动）
  ├── tokens 仍在（舰船数据保留）
  └── 仅 GM 可操作航行/事件

mode = COMBAT（进入战斗后）
  │
  ├── turn = { number: 1, factionIndex: 0 }
  ├── faction 由战斗预设决定
  ├── tokens 在战术地图上
  └── 战斗结束后返回 WORLD_EXPLORATION
```

这样世界观地图和战术战斗在状态机层面完全解耦，不需要共享 `phase`。
