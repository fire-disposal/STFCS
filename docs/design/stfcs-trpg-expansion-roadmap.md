# STFCS 向《远行星号》太空 TRPG 跑团平台扩展方案（v1）

> 目标：在不破坏 STFCS 现有“权威战术战斗内核”的前提下，将系统扩展为 **可长期战役、可叙事推进、可多人协同主持** 的远行星号风格太空 TRPG 平台。

---

## 1. 现有系统能力盘点（基于代码实现）

## 1.1 已具备的“战斗规则内核”优势

1. **权威服务器 + RPC namespace 分层**：服务器集中验证与状态推进，具备扩展更多规则模块的基础。现有 `edit/deploy/room/customize/game` 命名空间已经形成“规则操作入口分流”。
2. **核心数据模型统一于共享 schema**：`GameSchemas.ts` 已覆盖舰船、武器、护盾、辐能、玩家、阶段与阵营，且前后端共用，适合作为后续 TRPG 子系统（任务、角色、派系声望等）扩展锚点。
3. **战斗阶段与轮转已稳定简化**：`DEPLOYMENT -> PLAYER_ACTION` 的两阶段流程让主持节奏更可控，适合作为“战术场景层”的最小状态机。
4. **武器目标解算链条完整**：从挂载点世界坐标、射程、射界、目标象限到 UI 状态均可计算，为后续“火控官、传感、ECM、锁定”机制提供落地点。
5. **护盾/辐能系统可承载高决策密度玩法**：护盾开关、方向、覆盖、受击辐能化、维持开销等都已成型，是远行星号风味最核心的“攻防博弈资源”。
6. **几何与坐标已统一抽象**：`@vt/data/geometry` 把角度、坐标、扇形、移动向量统一，能够支撑更复杂战场对象（航迹、地形场、导弹群、跃迁门）。

## 1.2 已具备的“跑团工具化”能力

1. **DM 专属控制侧栏**：可强制推进阶段/回合、转移房主、踢人，说明系统已考虑主持人主权。
2. **玩家就绪与房间协作流**：TopBar/TurnBar + ready 状态使“多人同步节奏控制”已有实现基础。
3. **资产与头像体系**：已有 `asset:upload/list/batch_get` 与 Avatar 机制，可扩展到“角色立绘、战报图、任务简报图”。
4. **存档体系**：`save:create/list/load/delete` 与房间自动归档，天然支持战役长期推进（session 级归档）。

---

## 2. 当前短板（为何“战术游戏”还不够“TRPG 跑团”）

## 2.1 叙事层缺失

- 当前模型重心在舰船 token，**缺少角色卡、舰员分工、关系网、派系声望、任务日志**。
- DM 的判定流程主要是战斗推进，**缺少“非战斗场景”通用判定器**（社交、调查、后勤、维修、谈判）。

## 2.2 战役层缺失

- 存档存在但更像技术快照，缺少：
  - 战役时间线
  - 星区状态变化
  - 经济与补给
  - 舰队维护与改装进度

## 2.3 组织协作层缺失

- 单房主模型已能运行，但对于长战役常见的“联合主持（DM + 副官）”、“私密信息分发（秘密频道）”支持不足。

---

## 3. 目标产品定位（建议）

将 STFCS 划分为三层：

1. **战术层（已强）**：舰船战斗、火控、护盾、辐能、移动。
2. **叙事层（待补）**：角色行动、技能判定、事件推进、线索管理。
3. **战役层（待补）**：星图远征、补给维修、派系政治、战果持久化。

对应《远行星号》跑团体验：
- 玩家既操舰，也扮演“舰上岗位角色”。
- 一次 Session 在“桥段叙事”和“战术交战”之间切换。
- 每回合战斗结果影响长期战役生态（势力、资源、威望、危机）。

## 4. 扩展设计蓝图（代码实现 + 桌游机制 + DM 实战）

## 4.1 第一优先级：桥接“角色扮演”与“舰船战斗”

### A. 岗位制（Bridge Crew Roles）

**设计**：每位玩家绑定岗位（舰长、舵手、火控官、工程官、电子战官、通信官）。

**机制建议**：
- 每岗位每回合拥有 1~2 个“岗位动作点”。
- 岗位动作可修改战术层参数：
  - 火控官：临时提高某武器命中窗口 / 共享锁定
  - 工程官：紧急排散、局部修复、超频推进
  - 电子战官：干扰目标传感，降低敌方有效射程
- 与现有 `WeaponTargetingResult.uiStatus`、`shield`、`flux` 等字段直接联动。

**落地建议**：
- 新增 `CrewRoleSchema`、`CrewActionSchema`。
- 在 `room/action` 或 `game/action` 下新增 `crew:execute`。
- 在回合结束结算时统一应用岗位修正（避免并发冲突）。

### B. 非战斗判定器（Narrative Check Engine）

**设计**：为 DM 提供统一判定工具，支持属性+技能+情境修正+难度阈值。

**机制建议**：
- 采用 `2d10 + 能力值 + 专长 - 压力` 对抗 DC 的统一框架（或 d100 百分比，二选一）。
- 支持“推进轨（Progress Clock）”：例如“破解黑匣子 0/6”。
- 可选“代价成功”：成功但触发 heat/警戒值。

**落地建议**：
- 新增 `CharacterSchema`、`SkillCheckSchema`、`ClockSchema`。
- 客户端新增 `DM 判定面板` + `玩家请求判定`。
- 结果写入战役日志，形成可追溯叙事记录。

## 4.2 第二优先级：战役层（Campaign Loop）

### A. 星图远征与航线风险

**设计**：从“单张战斗地图”升级为“星区节点图 + 路线风险表”。

**机制建议**：
- 节点类型：港口、遗迹、风暴区、封锁带、黑市站。
- 每次跃迁触发风险检定（燃料、伏击、设备故障、情报机会）。
- 战术战斗成为节点事件的一种，而非唯一玩法。

**落地建议**：
- 新增 `SectorMapSchema`、`RouteEventSchema`、`CampaignStateSchema`。
- 复用现有 save/load；增加“战役时间推进”和“节点状态变更”记录。

### B. 经济/补给/维修闭环

**设计**：引入补给压力，让战斗结果与后续选择真正耦合。

**机制建议**：
- 资源池：补给、燃料、备件、情报、信誉。
- 战后流程：战损评估 -> 维修分配 -> 改装决策 -> 人员休整。
- 引入“机会成本”：紧急修复快但高代价、完整检修慢但安全。

**落地建议**：
- `TokenRuntime` 增加可持久化战损标签（模块损坏、临时修复状态）。
- `asset` 系统可存战报截图与维修报告作为战役档案。

## 4.3 第三优先级：远行星号风味机制增强

### A. 派系与声望博弈

**机制建议**：
- 对主要势力设 `-100 ~ +100` 声望。
- 声望影响商店库存、雇佣军质量、追捕强度、情报可见度。
- 与当前 `Faction`（战斗阵营）分离：
  - 战斗阵营 = 当前战场敌我
  - 声望阵营 = 战役关系网络

### B. 舰船模组/官能重构

**机制建议**：
- 把 `plugins` 扩展为“舰船改装件系统”（物流舱、相位稳定器、AI 核心、传感阵列）。
- 改装件提供：
  - 常驻效果
  - 主动技能（冷却）
  - 副作用（维护成本、故障率）

### C. 危机时钟与事件导演工具

**DM 经验向建议**：
- 每个战役同时跑 3 条隐藏时钟：
  - 追猎时钟（敌对势力定位你）
  - 政治时钟（地区冲突升级）
  - 内部时钟（船员士气/忠诚崩坏）
- 让 DM 面板可一键推进时钟并自动触发事件包。


## 4.4 补充：复杂度可控的“星区/战斗地图体系”初步方案

> 目标：避免“空旷平面图”，但也不引入过重的 4X 级模拟复杂度。采用 **两层地图 + 三类地形对象 + 统一效果标签** 的中等复杂度方案。

### A. 双层地图模型（战略层 + 战术层）

1. **星区层（Sector Layer）**
   - 表现为“节点 + 航线”图，而非连续大地图。
   - 只负责：移动成本、风险、补给机会、事件触发权重。
   - 单次 session 中变更频率低，适合战役推进。

2. **战斗层（Battle Layer）**
   - 表现为局部战场（如 3000x3000 或 4000x4000），承接现有 token 战斗系统。
   - 只负责：遮蔽、减速、传感干扰、危险区伤害等即时战术效果。
   - 单场战斗中高频计算，必须保持判定简单。

**设计关键**：星区层决定“打什么战”；战斗层决定“怎么打赢这场战”。

### B. 战斗地图对象：仅保留三大类（控制复杂度）

#### 1) 阻挡体（Obstacles）
- 小行星群、残骸带、空间站碎片。
- 规则：
  - 进入区域触发机动惩罚（转向率/速度上限下降）。
  - 穿越时可能吃到轻微擦伤（可选检定）。
  - 可提供“视线遮蔽等级”（影响远距武器或锁定）。

#### 2) 危险区（Hazards）
- 等离子风暴、辐射云、引力剪切区。
- 规则：
  - 回合结束结算固定效果（例如硬辐能增加、护盾效率惩罚）。
  - 强度分级（I/II/III），避免复杂公式。

#### 3) 资源/目标点（Objectives）
- 导航浮标、数据中继站、补给舱、跃迁信标。
- 规则：
  - 占领/交互可提供战术奖励（命中加值、临时补给、撤离胜利条件）。
  - 支持非歼灭胜利（护送、夺取、破坏、坚持回合数）。

### C. 统一“地形效果标签”系统（Terrain Tags）

为避免每种地形写一套独立逻辑，建议所有地图对象统一走标签：

- `BLOCK_LOS:x`（视线遮蔽强度）
- `SPEED_MULT:x`（速度倍率）
- `TURN_RATE_MOD:x`（转向修正）
- `FLUX_PER_TURN:x`（每回合辐能变化）
- `HULL_DMG_PER_TURN:x`（每回合船体伤害）
- `SENSOR_RANGE_MULT:x`（传感距离倍率）

这样服务端结算流程可固定为：
1) 收集 token 所在区域标签；
2) 合并同类标签（加法或乘法）；
3) 在回合结算统一应用。

### D. 地图生成与复用（DM 友好）

建议提供三种生成方式：

1. **模板地图**（推荐默认）
   - 例如：小行星伏击、风暴追击、跃迁门争夺。
   - 优点：平衡可控、上手快。

2. **参数化随机**
   - 输入参数：危险度、遮蔽密度、目标点数量。
   - 输出可重现（记录 random seed）。

3. **DM 手工编辑**
   - 放置对象并设置标签，不要求 DM 写规则脚本。

### E. 与现有系统的最低耦合落地点

1. 复用现有 `map` 概念，增加 `battlefieldFeatures` 数组。
2. 服务端在“移动后/回合结束”两个时机进行地形判定（不要每帧判定）。
3. 武器目标计算阶段增加“视线/传感修正”钩子，不改核心射界算法。
4. DM 面板增加“地图特性开关与强度调整”。

### F. 建议的首批最小可玩内容（MVP）

- 地图对象只做 6 个：
  - 阻挡体：`asteroid_field`, `wreckage_field`
  - 危险区：`plasma_storm`, `radiation_cloud`
  - 目标点：`relay_node`, `jump_beacon`
- 胜利条件增加 2 个：
  - `CONTROL_OBJECTIVE_N_TURNS`
  - `ESCORT_TO_BEACON`

这样可以在 1~2 个迭代内把战斗从“空场对射”升级为“地形决策博弈”，复杂度仍在团队可控范围内。

## 5. 技术演进路线（分期）

## Phase 1（2~4 周）：低风险高收益

- 增加战役日志（时间戳 + 事件类型 + 参与者 + 结果）。
- 增加岗位与岗位动作（先做少量动作：火控辅助、工程紧急散热）。
- 增加 DM 判定器（先做通用骰池 + 文本结果）。

**为什么先做**：几乎不破坏现有战斗链路，能立刻让“跑团感”提升。

## Phase 2（4~8 周）：战役闭环成型

- 星区节点图 + 路线事件。
- 经济/补给/维修三资源系统。
- 战后阶段 UI（Repair / Refit / Crew）。

## Phase 3（持续迭代）：内容工业化

- 势力包、事件包、任务模板、遭遇模板（数据驱动）。
- DM 剧本工具：可视化编排触发条件。
- 模组化扩展 API（社区内容可导入）。

## 7. 与现有 STFCS 架构的兼容原则

1. **战术内核不推倒重来**：所有新机制优先“附加”在现有 token/turn/rpc 上。
2. **Schema 先行**：延续 `@vt/data` 单点定义，前后端一致演进。
3. **事件日志化**：新增机制都要写入可回放日志，方便 DM 复盘与战报。
4. **DM 主权保留**：任何自动化都提供“DM override（覆写）”。

---

## 8. 作为跑团产品的“野心上限”建议

若目标不止“战棋工具”，而是“远行星号在线跑团平台”，可设三年愿景：

- **规则层**：战术+叙事一体化规则引擎（可切换规则模板）。
- **内容层**：官方事件包 + 社区工坊（舰船、任务、星图、派系包）。
- **运营层**：战役大厅、战报分享、观战回放、DM 信誉系统。

这条路线的关键不是“更多数值”，而是：

> 把每一次战斗，变成战役叙事中的一个“后果节点”。

当 STFCS 能稳定地产生“可追溯后果链”，它就会从优秀战术工具，成长为真正适合远行星号世界观的太空 TRPG 平台。
