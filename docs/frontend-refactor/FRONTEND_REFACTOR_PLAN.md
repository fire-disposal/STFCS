# 前端问题总结与改进计划（Pixi 一等公民）

日期：2026-04-15

## 1. 总体目标
- **Pixi 为一等公民**：游戏地图内的一切渲染与交互以 Pixi 为主导，其他层只提供数据与指令。
- **类型统一**：@vt/types 为唯一事实源，客户端不得自行扩展冲突字段。
- **边界清晰**：业务 UI / 游戏逻辑 / 渲染交互三层严格分离。

---

## 2. 主要问题总结

### 2.1 类型与模型混乱
- 客户端存在大量“自定义/兼容”字段，类型漂移严重。
- @vt/types 与客户端接口不一致，导致编译错误与运行时歧义。

### 2.2 逻辑分散
- **交互**：Pixi hooks 与 Redux hooks 并行，导致重复逻辑与行为不一致。
- **坐标/角度**：mathUtils、angleUtils、angleSystem 交错使用，缺少统一入口。
- **状态**：UI Store、Redux Store、Colyseus 状态并存但缺少分层规则。

### 2.3 地图内边界不清
- 地图内逻辑（选择、移动、拖拽、旋转）仍有部分在 React/Redux 层处理。
- Pixi 没有完全主导地图内交互与渲染。

---

## 3. 边界定义（必须遵守）

### 3.1 Pixi 负责
- 地图内渲染（所有 token、网格、特效、指示器）。
- 地图内交互（点击、拖拽、旋转、缩放、命中测试）。
- 坐标转换与相机变换。

### 3.2 React/UI 负责
- UI 面板、弹窗、列表、按钮等业务层交互。
- 将命令/输入传递给 Pixi（通过明确 API 或 Store）。

### 3.3 Store 负责
- 持有规范化后的**只读游戏状态快照**。
- Pixi 读取 Store，React UI 读取 Store。
- Store 不直接处理 Pixi 交互细节。

---

## 4. 改进计划（分阶段）

### 阶段 A：结构清理与约束（高优先级）
1. **移除双轨交互逻辑**：
   - 地图内交互统一由 Pixi hooks 处理。
   - Redux 交互层（旧 useInteraction 等）逐步下线。
2. **定义“地图内 API”**：
   - 例如 `MapInteractionAPI` / `MapCameraAPI`，由 Pixi 提供。
3. **强制边界规则**：
   - 地图内功能不得在 UI 层直接实现，只能通过 Pixi API 调用。

### 阶段 B：类型统一与规范化层（高优先级）
1. 引入 `normalize` 层，统一对 @vt/types 的适配：
   - `normalizeCameraState` / `normalizeShipState` / `normalizeTokenInfo`。
2. Store 只接受规范化结果。
3. 禁止新增“兼容字段”。

### 阶段 C：坐标与角度体系统一（高优先级）
1. 合并 `mathUtils` / `angleUtils` / `angleSystem`：
   - 统一为 `CoordinateSystem` 模块。
2. 所有 Pixi 变换、命中测试、平移/旋转/缩放都调用统一入口。

### 阶段 D：Pixi 主导渲染体系（中优先级）
1. 图层体系与更新策略完全由 Pixi 驱动。
2. React 仅提供渲染配置与外部 UI 控制。

### 阶段 E：验证与性能（中优先级）
1. 建立交互行为测试：
   - 旋转后平移方向正确。
   - 点击不触发拖拽，拖拽不触发点击。
2. 性能指标：
   - Pixi 渲染层帧稳定，避免 UI 触发过量重绘。

---

## 5. 执行建议（近期）
- **先完成交互双轨清理**，确保 Pixi 成为唯一入口。
- **再推进类型规范化层**，逐步消除兼容字段。
- **最后重构坐标/角度体系**，保证地图交互一致性。
