# STFCS 分阶段可用化分析（2026-03）

> 目标：聚焦“复杂度过高、基础流程不闭环、历史遗留问题”三类问题，给出可快速投入实际可用的分阶段方案。

## 1. 当前关键问题总览

### 1.1 系统复杂度过高（实现密度 > 当前交付能力）

1. **协议体系并行演进，存在双轨负担**：`shared/ws` 已有完整消息类型与请求响应 schema，同时还存在 `shared/protocol/MessageDSL` 与 `DomainEvents` 体系，导致团队需要维护两套概念与实现映射。  
2. **同一业务通路存在重复入口**：`MessageHandler` 同时处理 `REQUEST(ship.move)` 与直接 `SHIP_MOVED` 事件，两条路径都调用 `ShipService.moveShip`，增加行为分叉与排障复杂度。  
3. **客户端页面层承载过多状态与流程**：`App.tsx` 同时承担连接、入房、状态初始化与事件订阅；`GameView.tsx` 继续承接战斗/回合/工具交互，边界不清晰，后续功能越加越难拆。  
4. **大量 `as any`、placeholder 与 TODO 残留**：类型系统和功能边界都在“临时状态”，技术债已影响新功能速度。

### 1.2 基础游戏流程未闭环（可演示 ≠ 可玩）

当前更像“可联机画布 + 部分舰船规则”，并未形成最小可玩闭环：

- 进入房间后缺少“对局启动 → 回合推进 → 行动落地 → 持续叙事/中场控制”的强约束流程。  
- 回合服务存在，但并未在主服务初始化中接入统一调度，回合状态与移动/战斗状态脱节。  
- 战斗与移动仍偏“接口可调用”，缺“客户端强引导 + 服务端权威裁定 + 广播回写 + 日志审计”的一条龙闭环。

### 1.3 历史遗留问题（文档、架构、代码状态不一致）

1. **文档与代码栈认知漂移**：架构文档仍描述前端为 Vue/Pinia，但代码实际为 React + Redux Toolkit，导致新人理解成本与误操作风险上升。  
2. **身份模型存在临时化实现**：前端入场使用临时 `player_${Date.now()}` 作为 ID，服务端校验策略较弱，长期会在断线重连、权限校验、对局回放上出现一致性问题。  
3. **服务端启动方式耦合测试/导入场景**：`main.ts` 末尾“总是启动 server”，会给集成测试、脚本化复用和容器内多进程编排带来副作用。  
4. **领域表达存在语义混淆**：`ShipService.getShipStatus` 中 `hull` 由装甲总和计算，说明“舰体/装甲”语义边界尚未固定，后续战斗结算将被放大。

## 2. 机制与细节设计问题（逐机制拆解）

## 2.1 房间与连接机制

**问题**
- 连接与入房流程主要在前端拼装，服务端更像被动执行，缺少“会话生命周期状态机”。
- 断线重连的 player identity 与 room membership 恢复规则不完整。

**风险**
- 房间内“幽灵玩家”、重复玩家、状态错配。
- 线上排障难：不知道玩家到底是“未入房”“掉线未清理”还是“重连失败”。

**建议**
- 先定义 `session -> player -> room -> match` 四层生命周期，所有请求必须带 session/player 上下文。
- 增加服务端幂等接口：`player.reconnect`，而非重复 `player.join`。

## 2.2 回合机制

**问题**
- 回合结构完整，但与主流程未形成统一入口（战斗、移动、阶段切换没有单源驱动）。
- `nextTurn` 等逻辑存在，UI 可本地推进，但缺“服务端权威回合推进与广播锁”。

**风险**
- 客户端显示回合与服务端逻辑回合不一致，产生“我这边可行动/你那边不可行动”的体验割裂。

**建议**
- 回合推进只允许服务端产生（客户端仅发 `end_turn` 请求）。
- 增加 `turn.state.snapshot` 广播，客户端完全被动渲染。

## 2.3 移动机制（三段式）

**问题**
- 领域层已有 phase 校验，但 UI 侧缺强约束交互（A 段→转向→B 段）。
- 移动消息既有 request 也有 event 触发路径，且 room 绑定与 ship ownership 校验不完整。

**风险**
- 玩家可越权移动非本方单位；多端同时操作时会出现抖动与覆盖。

**建议**
- 锁定一条通路：客户端仅发 `request: ship.move`，服务端校验后统一广播 `SHIP_MOVED`。
- 每步校验必须包含：`playerId + roomId + shipOwner + turnPhase + remainingActionPoints`。

## 2.4 战斗机制

**问题**
- 伤害模型与武器判定存在，但缺“目标选择、命中结算、资源消耗、状态变更、回放日志”的完整交易。
- 舰体/装甲/护盾语义未完全定型，容易在数值上自相矛盾。

**风险**
- 出现“视觉命中但数值不扣”“日志显示不一致”等核心可信度问题。

**建议**
- 引入原子化 `combat.resolveAttack` 用例，单次请求返回完整结算对象（命中、每层吸收、最终伤害、触发事件）。
- 战斗日志结构化存储（最小先内存+环形 buffer），便于回放和仲裁。

## 2.5 状态同步与事件机制

**问题**
- 既有 WS message handlers，又引入 StateSync/EventBus integration，职责边界重叠。
- 客户端存在本地派发 + WS 事件回流并行，幂等控制薄弱。

**风险**
- 双写与重复渲染，状态“闪回”。

**建议**
- 为每类事件定义唯一 source-of-truth（建议服务端）。
- 增加 `eventId/sequence` 去重；客户端 reducer 按序列号忽略旧包。

## 2.6 UI/交互机制

**问题**
- `TacticalCommandPanel` 与 `GameView` 中仍有多个 placeholder/TODO，界面可见但行为未闭环。
- 关键动作（结束回合、护盾、散热、攻击）缺“可用性门禁”（哪些条件下可点击）。

**风险**
- 玩家误以为系统 bug（按钮能点但无效果）。

**建议**
- 先做“灰度可用”交互：不可用时明确置灰 + 原因 tooltip。
- 所有按钮动作统一走 request-response，失败提示必须显式。

## 2.7 工程治理机制

**问题**
- 文档偏旧、风格与架构漂移；技术债清单不集中。
- 代码中 `as any` 较多，导致重构时静态防线失效。

**建议**
- 建立单一“技术债登记板”（按影响/修复成本排序）。
- 先从高风险链路（连接、回合、移动、战斗）移除 `as any`。

## 3. 快速分阶段投入实际可用（90 天）

## 阶段 0（第 1-2 周）：收敛复杂度，冻结主干协议

**目标**：停止继续“加功能扩散”，先把主链路定成单轨。

**交付**
1. 冻结通信主协议（建议以 `shared/ws` 为主），明确 MessageDSL 为“后续演进层”或暂缓。
2. 统一 ship.move 单通路（仅 request 入口）。
3. 建立最小可观测性：请求日志、错误码、房间/玩家在线状态快照。
4. 修正文档与代码不一致项（至少架构栈、目录、术语）。

**验收标准**
- 新增功能 PR 不允许再新增并行协议路径。
- ship.move 线上只出现一种入口日志。

## 阶段 1（第 3-6 周）：打通“单局最小闭环”（可内测）

**目标**：实现“可完成一局简化战斗”的真实流程。

**闭环定义（MVP Match Loop）**
`加入房间 -> 选择单位 -> 回合开始 -> 三段移动 -> 一次攻击 -> 结算广播 -> 切换下一单位 -> 回合结束 -> 叙事继续`

**交付**
1. 服务端权威回合管理接入主流程。
2. 三段移动 UI 引导（步骤锁 + 预览 + 撤销当前步骤）。
3. 单武器攻击结算链路（含日志与动画最小反馈）。
4. 失败路径完善：越权、越阶段、非法参数都有明确错误码。

**验收标准**
- 4 人同房连续 30 分钟对局，不出现状态分叉。
- 所有关键动作均可在日志中复盘。

## 阶段 2（第 7-10 周）：从“能玩”到“好用”

**目标**：提高稳定性与操作体验，支持小规模公开试玩。

**交付**
1. 断线重连与会话恢复。
2. 关键 UI 的可操作性门禁与失败提示统一。
3. 战斗表现增强（命中反馈、伤害分层可视化）。
4. 回合超时/托管（防止对局卡死）。

**验收标准**
- 断线 30 秒内重连可恢复。
- 对局中断率显著下降（建议目标 <5%）。

## 阶段 3（第 11-13 周）：可运营化准备

**目标**：形成“可持续迭代”的基础。

**交付**
1. 数据留痕（对局回放最小事件流）。
2. 机制配置化（移动参数、武器参数、回合参数）。
3. 技术债清偿第一批（高风险 `as any`、主链路重构点）。

**验收标准**
- 能基于回放快速定位 80% 以上对局争议问题。

---

## 4. 优先级排序建议（按“上线价值/风险比”）

### P0（必须先做）
1. 协议与消息路径收敛（去双轨）。
2. 服务端权威回合 + 移动 + 攻击闭环。
3. 会话与身份模型稳定化（重连幂等）。

### P1（紧随其后）
1. 可观测性与日志回放。
2. UI 操作门禁与错误提示。
3. 文档与代码一致性治理。

### P2（可延后）
1. 高级特效与非核心视觉增强。
2. 扩展玩法（地形、战争迷雾、观战等）。

---

## 5. 一句话落地策略

**先把“一局打完”做成不可分叉的硬闭环，再谈机制扩展和体验升级。**

如果按该路径推进，项目可以在 4-6 周内达到“可内测可复盘”的实用状态，在 10 周左右进入“小规模公开试玩”阶段。
